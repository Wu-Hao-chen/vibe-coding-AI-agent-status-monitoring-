# AI Agent 状态监控面板

面向 AI 编程 Agent 的自托管状态监控面板。每个面板以实时红/黄/绿灯的形式展示 Agent 当前状态（空闲 / 执行中 / 等待权限批准），同时展示 Token 用量，并支持在网页上直接批准或拒绝一个待处理的权限请求。

| 项目 | 监控对象 | 后端架构 | 客户端集成方式 |
|------|---------|---------|---------------|
| [`claudestate/`](claudestate) | Claude Code | Node.js + Express（SSE 实时推送） | Claude Code 原生 Hook 机制 |
| [`hermesstate/`](hermesstate) | Hermes（一款桌面端 AI Agent） | Node.js + Express（SSE 实时推送） | PowerShell 脚本轮询进程状态 + 解析本地日志 |

面向 Codex 类 Agent 的第三个面板计划在后续版本中加入。

## 工作原理

三个面板遵循同一套设计：

1. **本地脚本 / Hook**：运行在 Agent 所在的机器上，监听其生命周期事件（工具调用开始/结束、出现权限请求、Agent 进入空闲），并通过 HTTP 上报给后端。
2. **轻量后端服务**（Node.js）：接收上报，在内存中维护一份当前状态，并通过 Server-Sent Events 将状态变化实时推送给所有已连接的浏览器。
3. **PHP 登录网关**：挡在面板前面，调用你自行实现的登录检查，通过后才会返回面板页面本身。
4. **浏览器端面板**：渲染红绿灯状态、最近活动记录和用量数据，并支持在页面上直接对一个待处理的权限请求做出批准/拒绝的决定。

### 红绿灯的含义

| 状态 | 含义 |
|------|------|
| 🟢 绿 | 空闲，当前没有工具调用在执行 |
| 🟡 黄 | 有一次工具调用正在执行 |
| 🔴 红 | Agent 被阻塞，正在等待权限批准的决定 |

Node 服务内置一个后台看门狗：如果某次工具调用超过 5 分钟没有收到任何后续活动，会自动把黄灯恢复为绿灯，避免因为漏发一个"已完成"事件导致灯永久卡住。红灯不受这个看门狗影响——它的生命周期由等待批准这个动作本身的超时机制决定，看门狗不应该抢先打断一个仍在等待用户做决定的请求。

### 实时推送的实现方式

Node 服务用 Server-Sent Events（`GET /events`）而不是 WebSocket——状态面板只需要单向推送，SSE 已经足够，而且浏览器的 `EventSource` 会自动处理断线重连，不需要额外实现。如果后端不方便维护一个常驻 Node 进程，也可以改为让前端定期轮询一个普通的 REST 接口，代价是实时性略差，换来的是一个更简单、无状态的后端实现。

## 安全模型

**这两个面板都没有内置身份验证。** 面板前面唯一的门槛，是你自己实现的一个函数——`auth_current_user()`，PHP 登录网关用它来判断当前请求是否已登录。除此之外，本仓库不对"谁能看到面板"和"谁能批准/拒绝一个待处理请求"做任何额外限制；Node 服务的 `/events` 和 `/decision/:id` 接口不会独立校验调用方身份。

这是刻意的设计取舍：本仓库是从一套私有部署中提炼、脱敏后发布的参考实现，那套部署原本使用的额外访问控制机制，与其具体运行环境高度绑定，不能安全地迁移到一份通用模板里——与其提供一份看起来完整、实际上无法安全复用的仿制版本，不如如实说明现状，交由使用者自行决策。

**在公开部署前，请评估这一风险并加入你需要的访问控制**，例如在 `auth_current_user()` 内加入角色/权限判断、引入二次验证、限制访问来源 IP，或将整个部署放在 VPN 之后。代码里对应需要处理的位置都留有注释标注。

## 目录结构

每个子项目目录下都有自己的 `README.md`，包含完整的部署步骤；`.env.example` / `config.example.php` 列出了该项目需要的每一项配置；代码里凡是与具体环境相关、需要替换成你自己的值的地方，都留有注释标注。

```
claudestate/
  server.js            Node 后端：接收 hook 上报、SSE 推送、处理批准/拒绝决定
  public/index.html     面板前端
  php/                  登录网关 + 面板页面的返回逻辑
  hooks/                需要安装到运行 Claude Code 的机器上的脚本
  .env.example

hermesstate/
  （目录结构与 claudestate/ 一致，客户端集成方式换成了轮询进程状态 + 解析本地日志）
```

## 已知限制

- **单一全局状态**：每个后端只维护一份状态，不区分具体是哪个 Agent 会话上报的。如果同时有多个会话上报到同一个部署，面板显示的是"最后一次上报"的状态，不会区分是哪一个。
- **不是开箱即用的成品**：这是一份参考实现，不是一个打磨好的安装包。你需要自己实现登录网关；`hermesstate` 里的日志解析正则是针对某个特定版本的 Agent 写的，换成别的 Agent 或者遇到新版本大概率需要自行调整。

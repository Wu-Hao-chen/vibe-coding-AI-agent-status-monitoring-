# state-dashboards

三个独立的 AI Agent 活动监控面板，各自监控一个不同的 Agent，用红/黄/绿灯 + Token 用量 + 远程审批展示"它现在在干嘛"。

| 项目 | 监控对象 | 后端 | 客户端集成 |
|------|---------|------|-----------|
| [`claudestate/`](claudestate) | Claude Code | Node.js + Express（SSE 实时推送） | Claude Code 原生 hook 机制 |
| [`hermesstate/`](hermesstate) | Hermes（桌面 Agent） | Node.js + Express（SSE 实时推送） | PowerShell 脚本轮询进程 + tail 日志 |
| [`codexstate/`](codexstate) | 你自己的 Agent | 纯 PHP + JSON 文件（前端轮询） | 你自己实现，按文档里的接口约定调用 |

每个子目录都有自己的 README，包含完整部署步骤。

## ⚠️ 没有内置验证——部署前必读

**这三套代码都没有内置的身份验证或权限控制。** 它们是从一套真实私有部署里拆出来的：原来那套有手机号+短信验证、TOTP 二次确认、角色/用户名白名单、PHP↔Node 之间的签名 cookie 校验等好几层机制，但全部跟原始部署的手机号、阿里云账号、用户名强绑定，没法安全地泛化成一份公开模板——把别人的示例密钥原样抄一份，比什么都不做还危险。所以这次全部拿掉了。

现在剩下的唯一门槛是：你自己实现的 `auth_current_user()` 有没有返回一个登录用户——**返回了就直接放行，不检查角色、不检查权限**。Node 服务的 `/events`、`/decision/:id`、`/api/state` 这些接口**完全没有认证**。

**部署前请自己想清楚风险，加上你自己需要的验证方式**：在 `auth_current_user()` 或各自的 `php/auth.php` 里加角色/权限检查、加二次验证、把整个服务放到 VPN 或 IP 白名单后面，都行——代码注释里标了具体该在哪里插入逻辑。CodexState 的 `CODEXSTATE_AGENT_TOKEN` 是例外，予以保留，因为它保护的是"Agent 侧上报凭证"这个跟部署强相关的动作，不是"用户登录后还要不要再验证"这类环节。

## 通用架构

三者都遵循同一套模式：

1. **Node/PHP 后端**接收本地脚本 POST 上来的状态，维护一个内存/文件里的当前状态对象。
2. **前端仪表盘**（SSE 推送或轮询）实时展示红/黄/绿灯 + 用量数据。
3. **PHP 登录网关**挡在仪表盘前面：检查你自己的上层账号系统，通过后直接进仪表盘。

## 这不是一个开箱即用的产品

这是一份去掉了所有真实密钥/域名/个人信息/验证逻辑的参考实现，不是一个"装好就能跑"的成品：

- 需要你自己有一个提供 `auth_current_user()` 的登录网关（本仓库不含），以及你自己决定要不要在这之上加更多验证。
- `hermesstate` 的日志正则是针对特定版本 Hermes 写的，换个 Agent 大概率需要自己重写这部分。
- `codexstate` 需要你自己写 Agent 侧的上报脚本。

如果你只是想抄一部分思路（比如 Claude Code 的 hook 集成方式，或者"红黄绿灯 + SSE"这套前端模式），单独看某个子目录即可，不需要整套部署。
